## Threat Model

Primary assets are live-order authorization, broker credentials, Supabase control-plane integrity, audit evidence, external provider data, and operator cockpit state.

### Assets

- Broker API credentials and live order authority
- Supabase service-role control-plane data
- manual_commands approval evidence and audit logs
- engine_events and alert webhook payloads
- external provider market/fundamental data

### Trust Boundaries

- Desktop publishable-key Supabase access versus worker service-role access
- Worker broker/provider adapters versus external APIs
- Supabase RLS/RPC boundary between anon/authenticated/service_role
- Worker event payloads crossing into external incident webhooks

### Attacker Capabilities

- Authenticated but non-admin Supabase user attempting to call exposed RPCs
- External provider or network fault returning oversized provider payloads
- Operator-controlled or event-derived metadata flowing into alert payloads

### Security Objectives

- Never place or enable live orders without fresh risk and manual approval evidence
- Fail closed on provider uncertainty and unknown order states
- Prevent secrets from reaching desktop, logs, docs, seeds, or webhook payloads
- Keep destructive Supabase operations restricted to the worker/service role

### Assumptions

- Current analysis is against the dirty working tree after remediation changes.
- Production assurance still requires hosted Supabase and provider/live drills.
